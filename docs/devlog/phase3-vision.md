### 트러블슈팅 ①: 시리얼 콘솔이 UART을 점유 (열화상 "안 보임"의 진짜 원인)

- **증상:** 열화상 모듈을 Pi의 GPIO14/15(기본 UART, `/dev/serial0 → ttyS0`)에
  RX/TX로 연결했는데 데이터가 잡히지 않을 상황. `/dev/ttyS0` 권한이
  `crw--w---- root tty` (dialout 아님)인 것도 단서.
- **원인 분석:** `/boot/firmware/cmdline.txt`에 `console=serial0,115200`이 있고
  `serial-getty@ttyS0.service`가 **active(enabled)** 상태였다. 즉 리눅스가 이
  UART을 **시리얼 콘솔(로그인 터미널)**로 쓰고 있었다. 이러면 두 가지가 깨진다:
  1) 부팅 시 커널 로그 + 로그인 프롬프트 바이트가 TX로 나가 **모듈로 쓰레기를
  쏘고**, 2) 모듈이 보내는 프레임을 **getty가 가로채** 애플리케이션이 못 읽는다.
  디바이스 노드가 `root:tty crw--w----`인 것도 agetty/console이 소유를 바꿔놓기
  때문.
- **해결:**
  - 런타임 해방: `sudo systemctl stop/disable serial-getty@ttyS0.service`.
  - 부팅 콘솔 제거: `cmdline.txt`를 백업(`cmdline.txt.bak.20260618`) 후
    `console=serial0,115200 ` 토큰만 sed로 제거(나머지 `console=tty1` 등은 유지,
    **cmdline.txt는 반드시 한 줄**이어야 하므로 줄바꿈 안 들어가게 주의).
  - 검증: getty 죽인 직후 `/dev/ttyS0`를 raw로 읽으니 115200에서 깨끗한
    `5a 5a ...` 프레임이 흘러나옴 → 점유가 풀렸음을 확인.
  - ⚠️ **남은 작업:** 디바이스 노드가 `root:tty`라 현재는 `sudo` 없이 못 읽는다.
    `cmdline.txt`에서 console을 뺐으므로 **재부팅하면** udev 기본 규칙에 의해
    `/dev/ttyS0`가 `root:dialout`로 돌아와 `aip1`(dialout 그룹)이 sudo 없이 접근
    가능해진다. 재부팅 전까지의 검증은 전부 `sudo`로 수행함.

### 트러블슈팅 ②: 열화상 UART 프로토콜 역공학 (`confirmed: false → true`)

- **상황:** 데이터시트가 없어 baud/프레임 포맷/체크섬이 전부 placeholder였다.
  맨 MLX90640 칩은 I2C 전용이므로, RX/TX로 나온다는 건 모듈에 **변환 MCU**가
  있어 그 MCU 고유 시리얼 포맷으로 프레임을 재송출한다는 뜻 — MLX I2C 레지스터와
  무관. 그래서 칩 데이터시트가 아니라 **선을 직접 떠서(empirical sniffing)**
  확정하는 게 유일하게 정확한 방법.
- **분석 과정(실무에서 미지의 시리얼 장치 다룰 때 그대로 쓰는 절차):**
  1. **baud 스캔:** 9600/115200/230400/460800/921600에서 1.5초씩 raw 캡처.
     115200만 깨끗한 구조(`5a 5a 02 06 cf 08 88 08 ...`)가 나옴. 나머지는
     프레이밍 깨진 노이즈(틀린 baud는 글자 깨짐처럼 보인다).
  2. **프레임 경계 찾기:** `5a 5a` 마커 위치들 사이 간격(gap)의 히스토그램을 보니
     **전부 1544바이트** → 고정 길이 프레임 확정.
  3. **payload 정렬:** 24×32=768px. uint16이면 1536B. `1544 = 2(마커) + 4(헤더)
     + 1536(payload) + 2(체크섬)` 조합만 **물리적으로 말이 되는 온도범위**
     (21.8–31.3°C)를 줬다. 다른 정렬(헤더 7/8)은 max 409°C·652°C 같은 헛값이라
     기각 — **"숫자가 물리적으로 말이 되는가"가 정렬 정답의 판별식.**
  4. **필드 해석:** 마커 뒤 `02 06`은 LE로 0x0602=1538=payload(1536)+csum(2),
     즉 **길이 필드**. 그 뒤 2바이트(offset 4–5)는 ×0.01 하면 ~22.4°C로
     payload 평균과 비슷 → **Ta(주변온도)** 필드.
  5. **체크섬 역공학:** sum8/CRC16(CCITT·Modbus)/바이트합 전부 불일치. 결국
     **bytes[0..1542)를 uint16 LE 워드로 보고 전부 더해 mod 0x10000** 하면
     트레일러와 일치 — **16프레임 연속 전부 일치**해서 우연이 아님을 확정.
- **반영:**
  - `thermal_config.yaml`: 실측값으로 채우고 `confirmed: true`. `header_bytes:
    4`, `value_byteorder: little`, `checksum: sum16_words` 필드 신설.
  - `thermal_serial.py`: 마커 뒤 헤더 스킵 + Ta 파싱(`last_ta_c`로 노출) +
    `sum16_words` 체크섬 검증 추가. (구버전은 마커 직후 바로 payload만 읽고
    1바이트 sum8만 알았음.)
  - 가드 철학 유지: `confirmed:false`면 `start()`가 여전히 거부 →
    검증 안 된 모듈로 바꾸면 조용히 가짜 온도를 만들지 않음.
- **자잘한 버그 2개(기록용):**
  - config Write 시 `start_marker` 키를 실수로 누락 → `KeyError`. 추가.
  - `np.uint64_scalar & 0xFFFF`가 numpy의 safe-casting 규칙에 막힘
    (`TypeError`). `int(...)`로 먼저 파이썬 int 변환 후 마스킹해서 해결.

### 트러블슈팅 ③: 카메라 media 노드 오설정 (`/dev/media1 → /dev/media0`)

- **증상:** `CameraCapture.start()`가 `media-ctl ... CalledProcessError`로 죽음.
- **원인:** `camera_config.yaml`의 `media_device: /dev/media1`이 틀림.
  `media-ctl -p`로 토폴로지를 보니 센서 `ov5647 10-0036`와 캡처노드
  `unicam-image(/dev/video0)`는 **`/dev/media0`**에 있고, `/dev/media1`은
  **bcm2835 ISP**(별개 미디어 그래프)였다. 엉뚱한 그래프에 센서 포맷을 set하려
  하니 실패.
- **해결:** `media_device: /dev/media0`로 수정. 이후 7초에 161프레임(~23fps),
  640×480×3, 평균밝기 119(소프트웨어 auto-gain 타겟 110 근처로 수렴) — 정상.

### 트러블슈팅 ④: 재부팅으로 권한 정상화 확인 (sudo 졸업)

- 재부팅 후 `/dev/ttyS0`가 `root:dialout`(이전 `root:tty`), `/dev/media0`가
  `root:video`로 돌아옴. `aip1`이 `dialout`·`video` 그룹 소속이라 **이제 sudo
  없이** 시리얼·카메라 접근 가능 — 트러블슈팅 ①에서 예고한 "재부팅하면 udev
  기본 규칙으로 복귀" 가설이 실측으로 확인됨.
- 검증: `thermal_serial`로 8/8 체크섬 통과, `camera_capture`로 640×480 30프레임
  연속 수신(평균밝기 122). 둘 다 `sudo` 없이.

### 트러블슈팅 ⑤: 열화상 죽은 픽셀(stuck pixel) — Phase 4 상시 오경보의 씨앗

- **증상:** 체크섬은 8/8 통과인데 프레임 `max`가 **615.4°C**로 튐(평균 28.4,
  Ta 24.9). 전송 오류가 아님 — **유효한(체크섬 OK) 바이트가 만든 비물리적 값**.
- **원인 분석:** 10프레임을 모아 좌표별로 보니 정확히 3개 픽셀
  `(10,28)·(13,2)·(23,14)`가 **항상 615.4°C(raw 61540), 프레임 간 std=0.0**으로
  고정. 장면 신호(이웃 픽셀은 24~36°C에서 출렁임)와 무관하게 못박혀 있음 →
  **MLX90640 센서 소자 자체의 죽은/고착(stuck) 픽셀**. MLX90640은 EEPROM에
  불량픽셀 목록을 갖지만, 우리는 모듈 MCU가 가공해 보내는 UART 프레임만 받으므로
  그 목록에 접근 못 함 → **수신측에서 휴리스틱으로 잡아야 함.**
- **왜 위험한가(severity 판단):** 단일 615°C 픽셀이 (a) colormap 정규화(자동
  레인지 쓰면)와 (b) **Phase 4 화재 임계 60°C**를 깨뜨려 **상시 ALARM_FIRE**를
  유발한다. "심한 문제"는 아니지만 방치하면 Phase 4가 무조건 깨지므로 지금
  파서에서 보정하는 게 옳다.
- **해결(추천 방식 자동 적용):** `thermal_serial._repair_bad_pixels()` 추가 —
  물리적 유효범위 `[min_valid_c, max_valid_c]`(`thermal_config.yaml`의 `bad_pixel`
  섹션, 기본 -20~300°C) 밖 픽셀을 **유효 8-이웃의 중앙값**으로 대체. 결과
  노출용으로 `last_bad_pixels`(직전 프레임에서 보정한 개수)도 제공. 적용 후
  8프레임 모두 `max≈35.8°C`로 물리값 회복, 매 프레임 3픽셀 보정 카운트 안정.
- **설계 선택 이유:** ① 보정을 **파서 단계**에 둠 → 모든 소비자(퓨전/AI/FSM)가
  깨끗한 프레임을 받음(중복 방어 불필요). ② **체크섬 검증과 분리** — 체크섬은
  "전송 무결성", bad-pixel은 "센서 결함"으로 책임이 다름. 체크섬을 통과한 값을
  굳이 버리지 않고 국소 보간만 함. ③ 임계값은 config로 빼서 다른 모듈/환경에서
  조정 가능.

### 트러블슈팅 ⑥: end-to-end 퓨전 첫 실하드웨어 구동

- 카메라(latest-frame-wins)·열화상을 동시에 띄워 `Fusion.overlay()`로 한 장 합성.
  출력 640×480×3 정상, 열화상 레인지 24.2~36.0°C, bad px 3개 보정됨.
- 화면이 전체적으로 청록 틴트인 건 **정상** — 상온 균일 장면(24~36°C)이 표시
  레인지 15~40°C에서 JET 저-중간(파랑/청록)으로 매핑되고 alpha 0.45로 블렌드되기
  때문. 강한 hotspot이 없으니 극적인 색이 안 나오는 게 맞다.
- ⚠️ **남은 사용자 작업(자동화 불가):** 정합 행렬 `affine`이 아직 identity라
  RGB↔열화상이 픽셀 단위로 안 맞는다. `pi/tools/fusion_calibrate.py`는 **가열
  타겟을 FOV에 흩뿌리고 두 창에서 같은 점을 클릭**하는 대화형 도구라 VNC 화면이
  필요 → 사용자가 실제 운용거리에서 직접 수행해야 한다. (카메라/열화상이 수직
  적층이라 시차가 거리의존적 — 운용거리에서 캘리브레이션 필수.)

### 트러블슈팅 ⑦: GUI 없는 서버 → 캘리브를 브라우저로

- **증상:** 기존 `fusion_calibrate.py`는 `cv2.imshow`+마우스 콜백 기반인데, 이
  머신은 Ubuntu **Server**라 X 디스플레이가 없어 `cv2.namedWindow`가 즉시
  실패(`Can't initialize GTK backend`). VNC도 안 떠 있음.
- **해결:** `fusion_calibrate_web.py` 신설 — 두 이미지를 브라우저에 띄워 클릭한
  점들을 POST로 받아 `estimateAffinePartial2D`로 affine 계산·저장. cv2의 highgui만
  X가 필요하지 `imencode`/`estimateAffinePartial2D`/`warpAffine`은 headless OK
  (기존 `camera_web_preview.py`와 동일 철학). 클릭 좌표계는 "RGB 크기로 업스케일한
  열화상 그리드"로, `Fusion.overlay`가 warpAffine 거는 그리드와 동일하게 맞춤.

### 트러블슈팅 ⑧: 고정 표시레인지가 열화상을 "안 변하는 것처럼" 보이게

- **증상:** 열화상 패널이 거의 균일한 청록색이라 손을 대도 변화가 안 보임.
- **원인:** `Fusion.colormap`은 고정 레인지 15~40°C. 상온(25~38°C)이 좁은 색
  구간에 몰려 콘트라스트 소멸. **딜레이가 아니라 대비 문제**였다.
- **해결:** 캘리브/라이브 표시에 한해 **프레임별 자동 대비**(2~98 퍼센타일 스트레치)
  + NEAREST 업스케일(픽셀 격자 또렷). **표시용일 뿐** — 저장 affine 기하나 실제
  Fusion의 고정레인지(화재임계 60°C)는 불변. (gray spread 좁음→15..226로 확장 확인.)

### 트러블슈팅 ⑨: 열화상 상하 반전 — **소스에서** 교정해야 하는 이유

- **증상:** 열화상이 카메라 대비 위아래가 뒤집혀 있음.
- **핵심 원리:** 캘리브 affine은 `estimateAffinePartial2D`=**similarity 변환
  (회전+등방스케일+이동)**이라 **뒤집기(반사)를 표현 못 한다.** flip을 교정 안 하면
  정합 자체가 수학적으로 불가능 → flip은 반드시 **캘리브 이전, 파서에서** 교정.
- **해결:** `thermal_config.yaml`에 `orientation`(flip_vertical/horizontal/
  rotate_180) 추가, `ThermalSerial.read_frame()`이 reshape 직후 flip 적용 →
  모든 소비자(라이브뷰·캘리브·Fusion)가 동일하게 교정된 프레임 수신(단일 진실
  소스). 표시단(`_render`)에 임시로 넣었던 flipud는 이중 뒤집힘 방지로 제거.

### 트러블슈팅 ⑩: 15~20초 지연 — tty 버퍼는 꽉 차면 **새 데이터를 버린다**

- **증상:** `Capture snapshot`이 15~20초 전 장면을 보여줌.
- **원인 분석:** 모듈은 연속 송신하는데 캘리브 도구는 캡처할 때만 1프레임 읽음 →
  OS 시리얼(tty) 입력 버퍼에 backlog. **결정적 사실:** 리눅스 tty 입력 버퍼는
  ~4KB(≈2.6프레임)로 캡되고, **꽉 차면 새 바이트를 드롭하고 오래된 것을 유지**한다.
  그래서 read_frame은 늘 옛 프레임을 읽고, 진짜 최신 데이터는 이미 버려진 상태.
  → backlog를 프레임 단위로 다 비워도(=drain) 전부 옛 프레임이라 소용없음.
- **1차 시도 실패:** `reset_input_buffer()`로 비우고 바로 읽음 → 스트림 **중간**에서
  재시작하니 `_sync_to_marker`가 픽셀 payload 안의 **가짜 `5A 5A`**에 잠겨
  ChecksumError(`want 0x025a`=다음 프레임 시작이 체크섬 위치에 옴)/FrameSyncError
  연쇄.
- **해결(`read_fresh`):** ① `reset_input_buffer()`로 stale 폐기(최신 데이터 통과
  시킴) ② read_frame을 **체크섬 검증+재시도**로 호출 — 가짜 마커에 잠겨도 실프레임이
  1544바이트 정렬이라 몇 프레임 안에 재동기화로 수렴(실측 0.15~0.4초). 연속
  리더(라이브뷰)는 버퍼를 늘 비워서 이 처리 불필요, read_frame 직접 사용.

### 트러블슈팅 ⑪(운영 교훈): `pkill -f <패턴>`의 자기 자신 매칭

- 백그라운드 서버 재시작이 간헐 실패. 원인은 `pkill -9 -f 'tools.fusion_calibrate_web'`
  가 **그 문자열을 명령줄에 담은 현재 셸까지 매칭**해 죽여서, 같은 명령 안의 launch
  라인이 실행 안 됨. **kill과 launch를 별도 명령으로 분리**하고, 포트 기준
  `fuser -k 8081/tcp`(패턴에 도구명 없음)로 정리해 해결.

### 트러블슈팅 ⑫: 와이드 카메라 + 점클릭 캘리브 → 과적합 → **라이브 수동 정렬**로 전환

- **증상:** 카메라가 **와이드 모듈**이라 점클릭 호모그래피 정합이 오버레이를 한쪽으로
  기울여 욱여넣음(저장된 H: scale 0.70, 이동 +91/+56, shear/회전 큼).
- **원인:** ① 와이드 렌즈의 perspective/edge skew, ② 열화상 24×32 저해상도라 손바닥이
  뭉개져 클릭 중심 오차 ±20~40px, ③ 호모그래피(8 DOF)가 이 클릭 노이즈를
  perspective 왜곡으로 **증폭(과적합)**. 점이 적거나 한쪽에 몰리면 더 심함.
- **중간 대응:** 캘리브 도구를 (a) 점을 **서버측 누적**(새로고침/모델변경에도 유지),
  (b) **모델 선택**(affine 기본 / similarity / homography) 가능하게 개선. affine(6 DOF)이
  저해상도 열화상엔 과적합 없이 종횡비/shear까지 잡아 최적. estimateAffine2D는
  **비연속(strided) 점 배열을 거부**(checkVector 단언) → `ascontiguousarray` 필요(버그픽스).
- **최종 해법(`fusion_align_web.py`):** 점 대응을 버리고 **라이브 수동 정렬**. 합성
  오버레이를 실시간 스트리밍하고 슬라이더 5개(scale X/Y·move X/Y·rotate)로 사람이
  눈으로 맞춤. 파라미터가 닮음+종횡비로 **제한**돼 과적합이 원천 불가, 손바닥 중심을
  정밀히 찍을 필요도 없음("핫블롭이 손 위에 올 때까지" 밀기). 최종 결과: 회전 0,
  scale 0.91/0.875, 이동 +54.8/+14 — 안정적. `captures/fusion_aligned.jpg`로 확인.
- **남은 한계:** 5-DOF라 배럴 왜곡(직선→곡선)은 못 잡음. 완벽히 필요하면 체스보드
  undistort가 정석. 추적/화재감지엔 중앙~중간 영역 정합으로 충분.

### 배운 개념 (공부용)

- **시리얼 콘솔 vs 주변장치 UART:** 라즈베리파이의 primary UART(GPIO14/15)는
  기본적으로 **리눅스 시리얼 콘솔**(부팅 로그+로그인)로 점유된다. 이걸 센서·모듈
  통신에 쓰려면 (1) `cmdline.txt`의 `console=serial0,...` 제거 + (2)
  `serial-getty@ttyS0` 비활성화, 두 군데를 다 풀어야 한다. 하나만 풀면 부팅 로그
  또는 로그인 프롬프트 중 하나가 여전히 선을 오염시킨다. 디바이스 노드 소유권이
  `root:tty`로 보이면 "아직 콘솔이 쥐고 있다"는 신호.
- **Pi의 두 UART (mini-UART vs PL011):** Pi4에서 `ttyS0`=mini-UART(코어클럭에
  baud가 묶여 살짝 불안정), `ttyAMA0`=PL011(안정적이지만 기본은 블루투스에 배선).
  `/dev/serial0`은 "현재 GPIO14/15에 배선된 쪽"을 가리키는 별칭이라, 보드 설정에
  의존하지 말고 이 별칭을 쓰는 게 안전.
- **미지의 시리얼 장치 역공학 절차:** ① baud 스윕으로 "구조가 보이는" 속도
  찾기 → ② 반복되는 sync 마커 간격으로 프레임 길이 확정 → ③ 알려진 차원
  (24×32×2B)으로 payload 정렬 후보 만들고 **물리적으로 말 되는 값**으로 정렬
  확정 → ④ 헤더 필드 의미 추론(길이/카운터/Ta) → ⑤ 체크섬은 후보군(sum8,
  sum16-byte, sum16-word, CRC16 변종)을 **여러 프레임에 동시에 맞춰** 판별.
  한 프레임에 우연히 맞을 수 있으니 **N프레임 연속 일치**가 확정 기준.
- **체크섬: 바이트 합 vs 워드 합:** 같은 데이터라도 8비트 단위로 더하느냐
  16비트 LE 워드 단위로 더하느냐가 다르다. 이 모듈은 후자(워드합 mod 0x10000).
  CRC가 아니라 단순 합이라 펌웨어 구현이 가볍고, 비트 뒤집힘은 못 잡지만
  프레이밍 어긋남(누락/삽입)은 충분히 걸러낸다.
- **Pi 카메라 미디어 파이프라인(libcamera 없는 환경):** Ubuntu Server엔
  `python3-picamera2`가 없어 raw V4L2 경로를 직접 운전한다. 센서 subdev·캡처
  노드·미디어 그래프가 **서로 다른 `/dev/mediaN`·`/dev/videoN`로 흩어져** 있어서,
  `media-ctl -p`로 토폴로지를 먼저 읽고 어떤 노드가 무엇인지 매핑하는 게 1순위.
  unicam(센서 raw)과 bcm2835 ISP는 별개 미디어 디바이스다. raw 경로엔 하드웨어
  ISP/3A가 없어 디베이어·화이트밸런스·auto-gain을 전부 소프트웨어로 처리.
- **MLX 모듈이 UART로 나오는 이유:** MLX90640 칩 자체는 I2C 전용. VIN/GND/RX/TX
  4핀으로 나온 모듈은 보드에 변환 MCU가 올라가 I2C로 센서를 읽어 **자체 시리얼
  프레임 포맷**으로 재송출하는 것 — 그래서 칩 데이터시트가 아니라 모듈의 출력
  포맷(우리가 역공학한 것)이 진짜 인터페이스다.
- **체크섬 OK ≠ 값이 정상 (transport vs sensor 결함의 분리):** 체크섬은
  "선을 타고 오는 동안 바이트가 안 깨졌나"(전송 무결성)만 보장한다. 센서 소자가
  죽어 항상 같은 헛값(615°C)을 내보내면 그 헛값까지 정직하게 체크섬에 포함돼
  **검증을 통과**한다. 그래서 결함 종류별로 방어 계층을 나눠야 한다: 프레이밍/전송
  = 마커+체크섬, 센서 결함 = 물리 유효범위·이웃 보간. 한 계층이 다른 계층의 일을
  대신해 줄 거라 가정하면 안 된다.
- **죽은 픽셀(dead/stuck pixel) 식별법:** "값이 크다"만으론 핫스팟인지 결함인지
  못 가린다. 판별식은 **시간적 불변성** — 여러 프레임을 쌓아 픽셀별 std를 보면
  진짜 장면은 출렁이고(작더라도 std>0), 죽은 픽셀은 **std=0으로 한 값에 못박혀**
  있다. 좌표가 매 프레임 동일하다는 것도 단서. 보정은 마스크 후 **유효 이웃의
  중앙값(평균 아님 — 또 다른 이상치에 강건)**으로 채운다.
- **affine(similarity) 변환이 못 하는 것 = 반사(flip):** `estimateAffinePartial2D`는
  회전+등방스케일+이동만 추정한다(4 DOF). 그래서 센서가 상하/좌우로 뒤집혀 있으면
  점 대응으로도 절대 못 맞춘다. **기하 보정의 순서가 중요** — flip/회전90°처럼
  좌표축 자체가 바뀌는 변환은 캘리브 *이전*에 소스에서 끝내고, 미세한 시차/줌은
  affine에 맡긴다. 일반화: "이산적 축 변환(flip/transpose)"과 "연속 변환(affine)"을
  파이프라인의 서로 다른 단계로 분리하라.
- **표시 정규화 ≠ 분석 정규화:** 같은 열화상이라도 *보기 좋게*(per-frame 자동대비)와
  *판정 기준*(고정 레인지, 화재임계 60°C)은 정규화가 달라야 한다. 자동대비를 분석에
  쓰면 임계가 장면에 따라 움직여 무의미해진다. 표시용 스트레치는 표시 코드에만,
  Fusion/FSM은 고정 레인지 유지.
- **연속 송신 센서 + 온디맨드 리더 = stale 함정:** tty 입력 버퍼는 작고(~4KB) **꽉
  차면 새 데이터를 드롭**한다(오래된 것 유지). 가끔 읽는 소비자는 늘 옛 프레임을
  본다. 버퍼를 다 비워 읽어도(drain) 그것들도 옛 것. 진짜 최신은 `reset_input_buffer`
  로 stale을 버리고 *새로 들어오는* 바이트를 읽어야 한다. 단 스트림 중간 재시작은
  프레이밍 재동기화(가짜 마커→체크섬 실패→재시도)가 필요. 연속 리더(백그라운드
  루프)는 버퍼를 늘 비워서 애초에 이 문제가 없다 — 캡처 패턴 선택이 곧 지연 설계다.
- **`pkill -f`의 자기 매칭:** `-f`는 전체 명령줄을 매칭하므로, 패턴 문자열을 담은
  현재 셸 자신도 걸려 죽을 수 있다. 서버 재시작 스크립트가 "조용히 launch를 건너뛰는"
  증상으로 나타난다. kill/launch를 분리하거나 포트 기준(`fuser -k`)으로 죽일 것.
- **DOF는 많을수록 좋은 게 아니다 (과적합):** 정합 모델의 자유도(similarity 4 →
  affine 6 → homography 8)가 클수록 표현력은 늘지만, **입력이 noisy/적으면 노이즈에
  맞춰 휜다.** 저해상도 열화상의 부정확한 클릭엔 자유도가 독. 모델 복잡도는 **데이터
  품질·양에 맞춰** 고르고, 의심되면 단순한 쪽 + reproj error로 검증. RANSAC도 점이
  몰려 있으면 못 구한다.
- **점대응 vs 라이브 수동 정렬:** 자동 점대응은 "정확한 대응점"이 전제다 — 그게
  어려우면(저해상도·뭉개짐) **사람을 루프에 넣는** 게 더 정확하고 빠르다. 의미 있는
  소수 파라미터(scale/translate/rotate)만 슬라이더로 노출하면, 사람은 "겹치는가"라는
  쉬운 판단만 하면 되고 모델은 구조적으로 과적합 불가. UI로 해 공간을 제약하는 것도
  정칙화(regularization)다.

### 트러블슈팅 ⑬: 정적 crop만으로는 부족 — 라이브 zoom 슬라이더로 확장

  ~70°로 좁혔다. 그런데 한 번 고정해버리면 **그보다 넓게 되돌릴 수 없다** —
  `CameraCapture`가 crop 적용 후의 프레임만 내놓기 때문.
- 해결: `CameraCapture.__init__`에 `force_full_frame` 옵션을 추가해, 캘리브레이션
  도구만은 저장된 crop을 무시하고 **항상 전체 센서 프레임**을 받게 했다. 그 위에서
  `fusion_align_web.py`가 매 프레임 "zoom"(crop 비율, 0.4~1.0)을 적용해 라이브로

  맞추는" 패턴을 적용한 것. **정적 설정과 라이브 조절 도구는 같은 데이터를 다른
  진실의 소스로 동시에 가지면 안 된다** — 도구 쪽이 항상 raw를 잡고, 배포 설정은
  도구의 "Save"가 써주는 결과물이어야 한다(역은 안 됨).

### 트러블슈팅 ⑭: "슬라이더가 안 먹는다" + "열화상 지연 심해짐" → 동시 실행 충돌

- **증상:** `fusion_align_web`(8083) 슬라이더를 움직여도 화면이 거의 안 바뀌고,
  열화상 프레임도 눈에 띄게 느려짐. 사용자는 "이전 버전이 더 부드러웠다"고 보고.
- **원인:** 디버깅 도중 `vision_web_preview`(8082)를 먼저 띄워둔 채로
  `fusion_align_web`(8083)을 추가로 띄웠다 — **두 프로세스가 동시에 같은
  `/dev/ttyS0`와 같은 카메라 디바이스를 열고 읽는** 상태였다. 시리얼은 멀티
  리더를 지원하지 않으므로 — 한쪽이 바이트를 읽어가면 다른 쪽은 그 바이트를 못
  보고, 프레이밍이 깨져 재동기화가 반복되며 지연·끊김으로 나타난다. 카메라도
  마찬가지로 V4L2 스트림 노드를 두 프로세스가 동시에 열면 경쟁이 생긴다. 슬라이더
  자체(`/set` 엔드포인트)는 정상이었지만, 화면이 거의 멈춰 있어 "안 움직인다"로
  보였을 뿐.
- **교훈 — 단일 리더 자원의 동시 실행 금지:** UART/카메라처럼 **배타적으로 열어야
  하는 디바이스**를 감싼 도구가 여러 개 있을 때, "도구 코드가 옳은가"를 보기 전에
  먼저 **"지금 몇 개가 동시에 그 자원을 잡고 있나"**부터 확인해야 한다. `ps aux |
  grep`으로 동일 자원을 다루는 다른 프로세스가 떠 있는지 항상 먼저 체크. 이 종류의
  버그는 코드를 한 줄도 안 고쳐도 "프로세스 하나 죽이기"로 해결되는데, 증상만 보면
  꼭 코드 버그처럼 보인다.
- **해결:** 정리 후 `fusion_align_web` 단독 실행으로 복귀, 정상 속도로 복원 확인.
  이후로는 카메라/열화상을 쓰는 도구(`vision_web_preview`, `fusion_align_web`,
  `app/main.py`)를 **항상 하나씩만** 실행하도록 운영 규칙 명시(코드 주석 +
  CLAUDE.md에는 미반영, 운영자 숙지 사항).

### 슬라이더 도구 확장 결과

- `fusion_align_web.py`에 **zoom 슬라이더**(camera FOV) 추가, 기존 5개(scale
  x/y, translate x/y, rotate)+alpha와 함께 총 7개 파라미터.
- **재진입성:** 도구를 다시 열면 `fusion_calib.yaml`의 저장된 warp를
  `sx/sy/tx/ty/rot`로 역분해하고 `camera_config.yaml`의 crop 너비로 zoom을
  복원해 슬라이더가 **마지막으로 저장한 값에서 시작**한다(매번 기본값으로
  리셋되지 않음) — `_decompose_warp()`가 도구가 만드는 정확히 그 형태의 행렬
  (`A = R(rot)@diag(sx,sy)`, center 피벗)이라는 가정 위에서 역연산.
- **Save가 두 파일에 쓴다:** `fusion_calib.yaml`(warp+alpha)과
  `camera_config.yaml`(crop 박스) 동시 갱신. 단 `camera_config.yaml`은 V4L2
  하드웨어 트러블슈팅 주석이 풍부해서 **`yaml.safe_dump` 전체 재작성 대신
  정규식으로 `crop:` 블록의 가치만 패치**(`_patch_camera_crop`) — 풀 덤프는
  주석을 전부 날려버리므로, "이미 검증된 운영 지식이 담긴 파일"과 "도구가 자동
  갱신하는 값"이 같은 파일에 있으면 자동 갱신 방식 자체를 보존 친화적으로
  골라야 한다.
- **최종 캘리브레이션(이 세션 종료 시점):** crop 512×384(zoom 0.8, 변경 없음),
  warp scale 1.115/1.25, translate 0.56/−82.0 (이전 0.91/0.875,
  54.8/14에서 와이드 crop 좁힌 후 재조정). 라이브 확인(`vision_web_preview`
  fusion pane)으로 열점이 RGB 물체 위에 정확히 겹치는 것 확인.
