# devlog — 개발 기록 컨벤션

이 폴더는 `PLAN.md`의 Phase 구분을 따라 **단계별 개발 일지**를 쌓는 곳이다.
목적은 두 가지: (1) 나중에 같은 트러블을 또 겪지 않기 위한 검색 가능한 기록,
(2) git에 올려서 진행 과정 자체를 공부 자료로 남기는 것.

## 파일 구조

```
docs/devlog/
├── README.md              # 이 파일 (컨벤션 설명)
├── phase1-bringup.md       # Phase 1: ESP32 펌웨어 + WiFi/MQTT 링크
├── phase2-kinematics.md    # Phase 2: IK/FK (해당 단계 시작 시 생성)
├── phase3-vision.md        # Phase 3: 비전/열화상 퓨전
└── phase4-ai.md            # Phase 4: 감시 AI/추적
```
다음 Phase에 들어갈 때 새 파일을 만들고, 이 README의 템플릿을 복사해서 쓴다.

## 항목 작성 템플릿

세션(작업한 날) 하나당 아래 형식으로 추가한다. 날짜는 항상 절대 날짜(YYYY-MM-DD)로.

```markdown
## YYYY-MM-DD

**목표:** 이 세션에서 하려던 것 한 줄.

**진행:**
- 한 게 뭔지 bullet로. 명령어/설정값처럼 재현 가능한 사실 위주로.

**트러블슈팅:**
- **증상:** 무엇이 안 됐는지 (에러 메시지, 관찰된 동작)
- **원인:** 왜 그랬는지
- **해결:** 무엇을 바꿔서 고쳤는지 (코드/명령/설정)

**배운 개념:**
- 이번에 새로 알게 된 개념/용어를 한두 줄로. (검색 키워드처럼 짧게)

**참고:**
- 링크, 데이터시트, man page, PLAN.md의 관련 섹션 등
```

트러블슈팅이 없었던 날은 그 섹션을 통째로 생략해도 된다. 매 항목을 다 채우려고
무리할 필요는 없음 — 핵심은 "다음에 검색해서 찾을 수 있는가"이다.

## 이 컨벤션 외에 실무에서 같이 쓰는 것들 (참고)

- **CHANGELOG.md** (Keep a Changelog 포맷) — devlog보다 압축된, "코드가 뭐가
  바뀌었는지"만 추적하는 릴리스 단위 기록. 이 프로젝트는 아직 릴리스 개념이
  없어서 생략했지만, Phase가 끝날 때마다 추가해도 됨.
- **ADR (Architecture Decision Record)** — 되돌리기 어려운 설계 결정을
  `docs/adr/0001-제목.md` 식으로 짧게 기록(배경/결정/대안/트레이드오프).
  예: "왜 Pi-ESP32 사이를 무선 MQTT로 분리했는가" 같은 결정이 후보.
- **Conventional Commits** — 커밋 메시지에 `fix:`, `feat:`, `docs:` 같은
  prefix를 붙이면 `git log`만으로도 변경 이력이 어느 정도 devlog 역할을 함.
- **GitHub Issues** — 트러블 하나당 이슈로 만들고 닫을 때 원인/해결을 코멘트로
  남기면, devlog 항목과 1:1로 링크할 수 있음(`docs/devlog`에는 요약만, 디테일은
  이슈 링크).
